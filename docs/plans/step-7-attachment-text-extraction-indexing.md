# Step 7 Implementation Plan: Attachment Text Extraction And Indexing

## Purpose

This plan covers step 7 from `docs/product-gap-analysis.md`:

> Add attachment text extraction/indexing.

The goal is to make supported attachment content searchable and retrievable by AI in the same permission domain as the parent article. Attachment matches should never bypass article visibility rules, and every result should make the source clear: article body or attachment content, including the attachment filename.

## Current Behavior

Relevant current behavior in this branch:

- `packages/db/src/schema.ts` defines `attachments` with file metadata only: `articleId`, `filename`, `storagePath`, `mimeType`, `sizeBytes`, and `createdAt`.
- Attachment upload, replace, delete, list, and download routes live in `apps/api/src/routes/attachments.ts`.
- Attachment files are resolved through `getUploadsDir()` and stored under `uploads/attachments/...`.
- `apps/api/src/services/import/import-engine.ts` creates attachment rows for imported Flowlu files but does not extract or index their content.
- Full-text article search uses `articles.search_vector`, populated from article title and `plain_text`.
- Semantic search and RAG use `article_embeddings`, generated by `apps/api/src/services/embedding-pipeline.ts`.
- `apps/api/src/services/search.ts` returns article-shaped results only.
- `apps/api/src/routes/rag.ts` returns article chunks only.
- `WorkspaceSearchResult` in `packages/types/src/index.ts` has no source fields for attachment matches.

Primary files to inspect before implementation:

- `packages/db/src/schema.ts`
- `packages/db/migrations/`
- `packages/types/src/index.ts`
- `apps/api/src/routes/attachments.ts`
- `apps/api/src/services/search.ts`
- `apps/api/src/services/embedding-pipeline.ts`
- `apps/api/src/services/embeddings.ts`
- `apps/api/src/routes/rag.ts`
- `apps/api/src/services/import/import-engine.ts`
- `apps/api/src/__tests__/routes/search.test.ts`
- `apps/api/src/__tests__/routes/rag.test.ts`
- `apps/api/src/__tests__/routes/attachments.test.ts`
- `apps/web/app/(main)/kb/[kbSlug]/search/page.tsx`

## Product Semantics To Implement

- Supported attachment types are extracted at minimum for PDF, DOCX, and plain text.
- Unsupported binary types remain downloadable but are marked `unsupported` for extraction and are not indexed.
- Extraction/indexing status is stored per attachment so the UI and operators can distinguish pending, processing, succeeded, failed, and unsupported files.
- Attachment full-text matches appear in normal search results when the parent article is visible to the user.
- Attachment semantic chunks appear in semantic search, hybrid search, and RAG retrieval when the parent article is published and in scope.
- Search and AI responses identify `sourceType: 'article' | 'attachment'`.
- Attachment matches include parent article metadata plus `attachmentId` and `attachmentFilename`.
- Replacing an attachment re-extracts and replaces all derived indexes for that attachment.
- Deleting an attachment deletes all derived text/index/embedding rows.
- Attachment indexing must not block a successful file upload longer than necessary. For the first pass, an in-process async job is acceptable, but the service boundary should allow moving work to a dedicated worker later.

## Reproduction Recipe

Before implementation:

1. Run `just dev`.
2. Log in at `/login` as `Local Admin` or `Local Editor`.
3. Create or open a published article in an accessible KB.
4. Upload a `.txt`, `.pdf`, or `.docx` attachment containing a unique phrase such as `marigold escrow remedy`.
5. Search the KB for `marigold escrow remedy`.
6. Observe no result appears unless the article body itself contains that phrase.
7. Call `POST /api/v1/rag/search` with the same query and an API key scoped to the KB.
8. Observe no chunk is returned from the attachment.

After implementation:

1. Repeat the upload with the same unique phrase.
2. Confirm the attachment row records extraction as `succeeded`.
3. Search the KB for the phrase in `fulltext`, `semantic`, and `hybrid` modes.
4. Confirm the result links to the parent article and shows that the match came from the attachment filename.
5. Replace the attachment with a new file containing a different unique phrase.
6. Confirm the old phrase no longer returns the attachment and the new phrase does.
7. Delete the attachment and confirm neither full-text search nor RAG returns it.
8. Log in as a user without article read access and confirm attachment-derived search results do not appear.

## Implementation Plan

### 1. Add Extraction And Indexing Schema

Add attachment extraction state to `packages/db/src/schema.ts` and a Drizzle migration:

- New enum `attachment_extraction_status`:
  - `pending`
  - `processing`
  - `succeeded`
  - `failed`
  - `unsupported`
- New nullable columns on `attachments`:
  - `extractionStatus`, default `pending`
  - `extractedText`
  - `extractionError`
  - `extractedAt`
  - `indexedAt`
  - `contentHash`
- Add an attachment `search_vector` tsvector column managed by a trigger from `filename` and `extracted_text`.
- Add a GIN index on the attachment search vector.

Add a dedicated semantic table instead of overloading `article_embeddings`:

```ts
export const attachmentEmbeddings = pgTable('attachment_embeddings', {
  id: uuid('id').primaryKey().defaultRandom(),
  attachmentId: uuid('attachment_id').notNull().references(() => attachments.id, { onDelete: 'cascade' }),
  chunkIndex: integer('chunk_index').notNull(),
  chunkText: text('chunk_text').notNull(),
  embedding: vector('embedding', { dimensions: 1536 }),
  createdAt: timestamp('created_at').notNull().defaultNow(),
});
```

Keep article and attachment embeddings separate so article re-embedding and attachment replacement can be updated independently.

### 2. Add Shared Source Types

Extend `packages/types/src/index.ts`:

- Add `SearchSourceType = 'article' | 'attachment'`.
- Add optional fields to `WorkspaceSearchResult`:
  - `sourceType`
  - `attachmentId`
  - `attachmentFilename`
  - `attachmentMimeType`
- Add the same source metadata to the RAG result type if one is introduced for machine clients.

Default existing article-body results to `sourceType: 'article'`.

### 3. Build Attachment Text Extraction Service

Create a focused service, for example `apps/api/src/services/attachment-indexing.ts`.

Responsibilities:

- Load the attachment and parent article.
- Resolve the stored file with the existing storage helper from `apps/api/src/routes/attachments.ts` or move that helper to a reusable module.
- Decide support by MIME type and extension.
- Extract text for:
  - `text/plain` and related text types using `fs.readFile`.
  - PDF using a server-side parser dependency.
  - DOCX using a server-side parser dependency.
- Normalize whitespace and cap extracted text length with an explicit env var such as `ATTACHMENT_EXTRACT_MAX_CHARS`.
- Store extraction status, text, error summary, extracted timestamp, and content hash.
- Delete old attachment embeddings before writing new ones.
- Chunk extracted text with the existing `chunkText()` helper.
- Generate embeddings with `createEmbeddingProvider()`.
- Insert rows into `attachment_embeddings`.
- Mark `indexedAt` after full-text and semantic indexing succeed.

Use a narrow error model:

- Unsupported type: status `unsupported`, no error log noise.
- Missing file: status `failed`, concise error.
- Parser failure: status `failed`, concise parser error.
- Empty extracted text: status `succeeded`, no embeddings.

### 4. Trigger Re-Extraction From Attachment Changes

Update attachment mutation paths in `apps/api/src/routes/attachments.ts`:

- On upload, create the attachment row with `extractionStatus: 'pending'`, then enqueue extraction.
- On replace, clear extracted text and old status, delete old `attachment_embeddings`, set `pending`, then enqueue extraction.
- On delete, rely on `attachment_embeddings` cascade and remove the attachment row as today.

For the first pass, use an internal async queue service inside the API process:

- `enqueueAttachmentIndexing(attachmentId)`
- `indexAttachmentNow(attachmentId)` for tests and backfills
- A small concurrency limit to avoid too many parser or embedding requests at once

Keep the queue in a module that can later be called by a worker service without changing route semantics.

### 5. Add Backfill/Reindex Entry Point

Add an operational script or API-internal command for existing imported attachments:

```sh
pnpm --filter @dovetail/api tsx src/scripts/reindex-attachments.ts
```

The script should:

- Find attachments with `articleId IS NOT NULL`.
- Skip unchanged rows when `contentHash` and `indexedAt` are current.
- Process pending, failed, or explicitly requested attachments.
- Support optional flags for one KB, one article, or one attachment if the existing script pattern supports arguments.
- Log counts for succeeded, unsupported, failed, and skipped.

This is required because Flowlu imports already create attachment rows before step 7 exists.

### 6. Include Attachments In Full-Text Search

Update `apps/api/src/services/search.ts` so full-text search runs over article body and attachment text.

Recommended shape:

- Keep the existing article query for `articles.search_vector`.
- Add an attachment query joining:
  - `attachments att`
  - `articles a`
  - `categories c`
  - `knowledge_bases kb`
  - `users u`
- Reuse the same `buildSqlFilters()` permission/status/filter logic against the parent article.
- Add `att.search_vector @@ websearch_to_tsquery(...)`.
- Return the same parent article metadata as article results.
- Set:
  - `sourceType: 'attachment'`
  - `attachmentId`
  - `attachmentFilename`
  - `attachmentMimeType`
  - `snippet` from `ts_headline('english', COALESCE(att.extracted_text, ''), ...)`
- Combine and rank by source rank, with article body matches preferred when scores are otherwise equal.

For pagination, count distinct source rows, not just distinct articles, so one article can legitimately have both an article-body match and an attachment match.

### 7. Include Attachments In Semantic And Hybrid Search

Update semantic search in `apps/api/src/services/search.ts`:

- Keep the existing `article_embeddings` query.
- Add an `attachment_embeddings` query joined through attachments to the parent article.
- Apply the same parent article filters.
- Return attachment source metadata.
- Merge article and attachment semantic rows by vector distance.

Update hybrid search:

- Use reciprocal rank fusion over source identities, not just article IDs.
- Recommended key: `article:<articleId>` and `attachment:<attachmentId>:<chunkIndex>`.
- Preserve the parent article link while showing the matching source.

### 8. Include Attachments In RAG Retrieval

Update `apps/api/src/routes/rag.ts`:

- Search both `article_embeddings` and `attachment_embeddings`.
- Scope attachment chunks through their parent published article and KB/category filters.
- Apply the same API key KB authorization.
- Return source metadata:
  - `sourceType`
  - `articleId`
  - `articleTitle`
  - `articleUrl`
  - `categoryPath`
  - `attachmentId`
  - `attachmentFilename`
  - `chunkText`
  - `score`
  - last edited metadata from the parent article

Keep full article content out of RAG search responses. Attachment chunks should be citation material, not a download substitute.

### 9. Update Search UI

Update `apps/web/app/(main)/kb/[kbSlug]/search/page.tsx`:

- Use the shared `WorkspaceSearchResult` type instead of a local duplicate if practical.
- Show a compact source label for each result:
  - `Article body`
  - `Attachment: <filename>`
- Keep the result link pointing to the parent article.
- Keep snippets plain text as today, but avoid stripping source metadata.

No attachment-specific search filters are required for step 7 unless product asks for them later.

### 10. Add Focused Tests

Add API service tests:

- Text extraction succeeds for `.txt`.
- Unsupported binary file sets `unsupported`.
- Missing file sets `failed`.
- Replacing an attachment deletes old attachment embeddings and writes new ones.
- Empty extracted text does not call the embedding provider.

Add search route tests:

- Full-text search returns an attachment match with `sourceType: 'attachment'`.
- Full-text search does not return attachment matches for unreadable parent articles.
- Semantic search returns attachment chunks.
- Hybrid search can include both article and attachment results without collapsing by article ID.
- Deleted/replaced attachments no longer appear.

Add RAG tests:

- RAG returns attachment chunks when the API key has KB access.
- RAG rejects requested KBs outside the API key scope as it does today.
- RAG response includes parent article metadata and attachment filename.

Add a small UI test if the current web test harness covers the search page:

- Attachment search results render the attachment source label and still link to the parent article.

## Rollout Notes

- Add extraction dependencies deliberately and pin them through `pnpm-lock.yaml`.
- Treat parser output as untrusted text. Do not render extracted text as HTML.
- Avoid logging extracted attachment content because attachments may contain sensitive legal material.
- Keep extraction errors concise in the database; detailed stack traces belong in logs only.
- If embedding generation is disabled or unavailable in local development, extraction and full-text indexing should still work.
- Consider exposing extraction status in the editor attachment manager after the backend fields exist, but do not make that UI a blocker for backend indexing.

## Out Of Scope For Step 7

- DOCX-to-rich-text article conversion. Step 7 extracts DOCX text for indexing only.
- OCR for scanned PDFs or images.
- Spreadsheet extraction.
- MCP endpoints and LibreChat documentation.
- A separate worker service. The code should make one possible later, but this step can run jobs inside the API service.
- Production backup/restore changes for uploaded files.
