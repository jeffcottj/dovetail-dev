# User Testing Issues — 2026-03-21

## Issue 1: Dark mode login page uses full-color logo instead of white
- **Step ID:** 2.1.4
- **Severity:** minor
- **Screenshot:** [`screenshots/2.1.4-dark-mode-login-fullcolor-logo.png`](screenshots/2.1.4-dark-mode-login-fullcolor-logo.png)
- **Expected:** MLA logo appears in white on dark background
- **Actual:** Logo src is `mla-secondary-fullcolor.png` (full-color) instead of white variant
- **Console errors:** `Failed to load resource: the server responded with a status of 400 (Bad Request)` for `/logos/mla-secondary-white.png`
- **Likely cause:** The `next/image` optimization endpoint returns 400 for the white logo asset. Either the file is missing from `public/logos/` or the image config in `next.config` doesn't cover it.

## Issue 2: Admin dashboard headings use DM Sans instead of Cardo serif
- **Step ID:** 2.1.8
- **Severity:** minor
- **Expected:** Section headings (h1, h2) on admin pages use serif font (Cardo)
- **Actual:** All admin headings use `"DM Sans", "Helvetica Neue", sans-serif`. Cardo is correctly applied to article editor headings (h2/h3 inside Tiptap) but not to admin page headings.
- **Likely cause:** The Cardo font-family CSS variable is only applied within the `.tiptap`/`.ProseMirror` editor scope, not globally to admin section headings.

## Issue 3: Sidebar collapse state not persisted across page reload
- **Step ID:** 2.2.5
- **Severity:** major
- **Screenshot:** [`screenshots/2.2.5-sidebar-collapse-not-persisted.png`](screenshots/2.2.5-sidebar-collapse-not-persisted.png)
- **Expected:** Sidebar remains collapsed after page reload (persisted via localStorage)
- **Actual:** Sidebar reverts to expanded state after reload. The collapsed state is not persisted.
- **Likely cause:** The sidebar collapse state is managed in React state only and not written to/read from localStorage.

## Issue 4: Post-save redirect uses wrong URL (missing category slug)
- **Step ID:** 2.4.2
- **Severity:** major
- **Expected:** After saving a new article, redirect to `/articles/<category-slug>/<article-slug>`
- **Actual:** Redirects to `/articles/<article-slug>` (no category prefix), which returns 404. The article exists at the correct URL with the category prefix.
- **Likely cause:** The `router.push()` after article creation builds the URL from only the article slug, omitting the category path. Same issue recurs after "Move to category" (step 2.4.11) — the page URL retains the old category prefix.

## Issue 5: Version restore does not work
- **Step ID:** 2.4.9
- **Severity:** major
- **Expected:** Clicking "Restore" on a previous version reverts the article to that version's content/title. A toast confirms restoration.
- **Actual:** After clicking "Restore" on Version 1 (original title "Test Article Alpha"), the article title remains "Test Article Alpha — Edited". No toast notification. No new version created. The restore appears to have no effect.
- **Likely cause:** The restore API call may be silently failing, or the frontend isn't calling the restore endpoint correctly.

## Issue 6: No status badge on article view page
- **Step ID:** 2.3.2
- **Severity:** minor
- **Expected:** A status badge (e.g., "Published") is visible on the article view page
- **Actual:** No badge elements found on article detail page. Status badges only appear in list views (landing page, category listings).
- **Likely cause:** The article view page template does not render a `<Badge>` component for the article status.

## Issue 7: Search missing date range and tag filters

- **Expected:** Search filter panel includes date range filter, tag filter, and "Clear all filters" button
- **Actual:** Filter panel only has search mode toggle and category dropdown. No date range, tag filter, or clear-all button.

## Issue 8: Tag badges on article view are not clickable links
- **Step ID:** 2.6.9
- **Severity:** minor
- **Expected:** Clicking a tag badge on an article navigates to `/search?tags=...`
- **Actual:** Tags are rendered as plain text/span elements, not as clickable links to search.
- **Likely cause:** Tag badges use `<span>` or `<div>` instead of `<a>` or `<Link>` wrapping.

## Issue 9: Category permission overrides not reflected in article view UI
- **Step ID:** 3.2.4
- **Severity:** major
- **Expected:** A viewer with an "editor" category-level override on Housing should see Edit/Actions buttons on articles in that category.
- **Actual:** After assigning the override, the viewer still sees no Edit button on Housing articles. The override appears to work at the API level (assigned/removed correctly in admin UI) but the frontend article view page does not check category-level roles.
- **Likely cause:** The article view page checks only the user's global role, not category-level overrides, when deciding whether to render Edit/Actions buttons.

## Issue 10: Import deduplication not implemented
- **Step ID:** 3.4.1–3.4.5
- **Severity:** major
- **Screenshot:** [`screenshots/3.4-reimport-no-dedup.png`](screenshots/3.4-reimport-no-dedup.png)
- **Expected:** Re-importing the same ZIP should detect duplicate articles/categories and report errors or skip them.
